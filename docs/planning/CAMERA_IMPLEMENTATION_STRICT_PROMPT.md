# カメラ機能実装 指示書（厳密版）

このドキュメントは、既存アプリに対して「カメラ起動からバーコード読取まで」を再現性高く実装させるための厳密プロンプトです。別AIにそのまま渡して実装を依頼できます。

## 1. そのまま使える厳密プロンプト

以下を一字一句そのまま渡してください。

---

あなたは既存Webアプリの改修担当です。
対象は Go + Gin + SQLite + HTML/CSS/Vanilla JS の在庫管理アプリです。

目的:
左側タブの「カメラ」で、PCカメラを使ったバーコード読取を完成させる。
「起動できる」だけでは不十分で、「読み取って既存検索フローに接続する」ところまで必須。

前提:
- 既存の「機器一覧」機能（カテゴリ/メーカー/製品選択、在庫表示、入出庫登録）を壊さない。
- 既存APIは変更しない（特に `/api/barcode/search` は現状利用）。
- 変更対象は主に `index.html`, `style.css`, `script.js`。
- 最小差分で実装する。

必須仕様:

1) タブ仕様
- 左上タブは `カメラ` と `機器一覧` の2つ。
- `カメラ` タブ選択時のみカメラペインを表示。
- `機器一覧` タブ選択時は既存の一覧UIを表示。

2) カメラ起動/停止
- `カメラ` タブ選択で `navigator.mediaDevices.getUserMedia({ video: { facingMode: 'environment' }, audio: false })` を実行。
- タブ離脱時は必ず `MediaStreamTrack.stop()` で停止し、`video.srcObject = null` を実施。
- `beforeunload` でも停止。

3) 読取機能（最重要）
- `BarcodeDetector` が利用可能なら、カメラ映像に対して継続検出ループを実行。
- 検出した `rawValue` を既存の正規化処理に通す（既存の normalize 関数を利用可）。
- 正規化した値をバーコード入力欄へ反映。
- 反映後、既存のバーコード検索処理を呼び出し、製品選択フローへ接続する。
- 同一バーコード連続検出の誤爆を防ぐためクールダウン（例: 1000-2000ms）を入れる。

4) テンキー表示ルール
- `カメラ` タブ選択やフォーカスでテンキーは出さない。
- テンキーはバーコード入力欄 `focus` 時のみ表示。
- タブ切替時にテンキーが表示中なら閉じる。

5) 起動時省電力ルール
- アプリ起動直後はカメラ・テンキー・一覧データ読み込みを必要最小限にする。
- ユーザー操作（タブ選択や入力フォーカス）をトリガーに起動する。
- 自動フォーカス等で勝手に起動しないようにガードする。

6) UI/状態表示
- カメラペインに以下を表示:
  - タイトル（例: カメラ読み取り）
  - 状態テキスト（待機中 / カメラ起動中 / 検出中 / 失敗理由）
  - videoプレビュー領域
  - 未起動時プレースホルダ
- カメラ拒否時は `NotAllowedError` など失敗理由を明示する。

7) 互換性
- `BarcodeDetector` 非対応端末では「自動読み取り非対応」メッセージを表示し、カメラプレビューのみ継続できるようにする。
- 例外で画面全体が落ちないこと。

受け入れ基準（必須）:

A. カメラタブ選択でカメラ起動し、機器一覧タブへ戻ると停止する。
B. バーコード読取成功時、入力欄に値が入り、既存検索が実行される。
C. 読取後に対象製品が選択状態へ遷移する（既存UI連携）。
D. カメラタブでテンキーは表示されない。
E. 入力欄 focus 時のみテンキー表示。
F. 起動直後にカメラ・テンキーが勝手に起動しない。
G. 拒否/失敗時にユーザーが理解できる文言が表示される。
H. JS構文エラー・ランタイム例外を出さない。

検証手順（実施して結果を報告）:

1. 構文チェック
- `node --check script.js`

2. カメラ許可ありケース
- カメラタブを選択
- ステータスが起動中になる
- 実バーコードを映す
- 入力欄へ反映される
- 既存検索が走り、製品情報が表示される

3. カメラ拒否ケース
- 権限拒否状態でカメラタブを選択
- 失敗理由（NotAllowedError等）がステータス表示される
- 画面全体は操作継続可能

4. テンキー確認
- カメラタブ選択時: 非表示
- バーコード入力欄 focus: 表示
- タブ切替時: 閉じる

5. 後始末確認
- タブ離脱/ページ離脱でカメラ停止（トラック停止）

出力形式:
- 変更ファイル一覧
- 変更点要約
- 受け入れ基準 A-H のチェック結果
- 残課題（あれば）

---

## 2. 依頼時の注意（運用者向け）

- 「実装だけ」ではなく、必ず「検証結果まで出力」を要求してください。
- 受け入れ基準 A-H を1項目ずつ `OK/NG` で報告させてください。
- 端末権限によって失敗する可能性があるため、許可あり/拒否の両ケースを確認させてください。

## 3. 期待成果物

- コード修正（HTML/CSS/JS）
- 実行ログまたは検証ログ
- 受け入れ基準チェック表

## 4. NG例（再現性を落とす指示）

- 「カメラを起動して」だけの曖昧指示
- 読取後の既存検索連携を未指定
- テンキー表示条件を未指定
- タブ離脱時の停止を未指定
